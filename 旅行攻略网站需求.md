# 旅行攻略网站需求

## 1. 项目目标

开发一个面向移动端的北京旅行攻略网站，帮助用户快速浏览北京知名旅游景点、查看基础攻略信息，并完成简单的行程参考。

项目定位为前端练手型 Demo，不追求复杂业务，不接入真实后端，使用 `Vue 3 + Mock` 模拟数据请求与页面展示。

## 2. 项目范围

本项目仅实现一个简单的移动端 H5 网站，包含以下核心能力：

1. 首页展示北京旅行主题信息与热门景点。
2. 景点列表页展示知名景点的基础攻略信息。
3. 景点详情页展示景点介绍、开放时间、门票、交通与游玩建议。
4. 推荐行程页展示 1 日或 2 日简单路线建议。
5. 使用本地 Mock 数据模拟接口，不接入数据库和真实服务端。

不包含以下内容：

1. 用户登录、注册、收藏同步。
2. 在线购票、支付、订单系统。
3. 地图导航、定位、评论发布。
4. 后台管理系统。

## 3. 目标用户

目标用户为准备到北京旅游、希望快速获取基础攻略信息的移动端用户，主要需求如下：

1. 快速了解北京值得去的景点。
2. 用较少时间查看每个景点的基本信息。
3. 获取简单直观的行程建议。

## 4. 技术方案

### 4.1 前端技术

1. 框架：`Vue 3`
2. 构建工具：`Vite`
3. 路由：`Vue Router`
4. 状态管理：可选 `Pinia`
5. 样式方案：`CSS` / `SCSS`，优先移动端适配
6. Mock 方案：`mockjs` 或本地 `JSON + Promise` 模拟请求

### 4.2 适配要求

1. 以移动端为主，推荐设计宽度参考 `375px`
2. 页面在常见手机宽度下正常显示
3. 不要求复杂响应式断点，但需保证桌面浏览器预览时布局不破碎

## 5. 页面需求

## 5.1 首页

### 页面目标

让用户一进入网站就能感知这是一个北京旅行攻略站，并快速进入景点浏览。

### 页面内容

1. 顶部标题区
2. 北京旅行主题 Banner
3. 热门景点卡片区域
4. 推荐行程入口
5. 底部导航

### 功能要求

1. 顶部显示网站名称，如“北京旅行攻略”
2. Banner 展示一句简短文案，例如“第一次来北京，这些地方值得去”
3. 热门景点区域展示 4 到 6 个重点景点
4. 点击景点卡片可进入景点详情页
5. 点击“查看全部景点”进入景点列表页
6. 点击“推荐行程”进入行程页

## 5.2 景点列表页

### 页面目标

集中展示北京主要景点，便于用户快速筛选和浏览。

### 页面内容

1. 顶部返回栏
2. 分类筛选区
3. 景点列表区

### 功能要求

1. 支持按分类筛选景点
2. 分类建议包含：`历史文化`、`皇家园林`、`现代地标`、`亲子休闲`
3. 每个景点卡片展示：
   - 景点名称
   - 封面图
   - 简短介绍
   - 推荐游玩时长
   - 门票信息
4. 点击卡片进入景点详情页

## 5.3 景点详情页

### 页面目标

详细展示单个景点的旅行攻略信息。

### 页面内容

1. 景点主图
2. 景点名称与标签
3. 基础信息区
4. 景点介绍
5. 游玩建议
6. 交通信息

### 功能要求

1. 展示景点名称、评分标签或推荐标签
2. 展示开放时间、门票价格、建议游玩时长
3. 展示景点详细介绍
4. 展示最佳游览季节
5. 展示交通建议，如地铁线路或公交说明
6. 展示游玩提示，如“建议上午前往”“周末人流较大”

## 5.4 推荐行程页

### 页面目标

给用户提供简单可参考的北京游玩路线。

### 页面内容

1. 一日游推荐
2. 两日游推荐

### 功能要求

1. 每条路线包含标题、适合人群、景点顺序、简短说明
2. 每个路线节点可展示对应景点名称
3. 可简要描述线路逻辑，例如“地理位置集中，适合首次来北京”

## 5.5 底部导航

建议采用 3 个入口，保持简单：

1. 首页
2. 景点
3. 行程

## 6. 景点内容范围

建议首版只放 6 到 8 个北京知名景点，控制内容规模，便于快速完成。

推荐景点如下：

1. 故宫博物院
2. 天安门广场
3. 八达岭长城
4. 颐和园
5. 天坛公园
6. 南锣鼓巷
7. 鸟巢
8. 什刹海

## 7.4 Mock 接口建议

建议模拟以下接口：

1. `GET /api/spots`
   返回全部景点列表
2. `GET /api/spots/:id`
   返回单个景点详情
3. `GET /api/hot-spots`
   返回首页热门景点
4. `GET /api/routes`
   返回推荐行程列表

## 8. 页面交互需求

交互保持轻量，避免过度设计。

1. 首页卡片点击跳转详情页
2. 列表页点击分类时切换筛选结果
3. 列表页和详情页支持返回上一页
4. 页面首次加载时展示简单 loading 状态
5. Mock 请求失败时展示基础错误提示

## 9. UI 设计要求

整体风格保持简洁、清爽、适合旅游主题。

1. 色彩建议偏中国风或城市旅行风格
2. 主色可使用红色、金色、米白色等与北京文化气质贴近的颜色
3. 卡片圆角适中，页面层次清晰
4. 字体大小适配手机阅读
5. 首页 Banner 不需要轮播，静态视觉即可

## 10. 非功能要求

1. 页面打开速度快，避免复杂动画
2. 所有数据均来自 Mock，保证本地可独立运行
3. 代码结构清晰，便于后续扩展真实接口
4. 图片可先使用占位图或网络示意图，后续再替换

## 12. 开发任务拆分

### 第一阶段：项目初始化

1. 使用 `Vite` 创建 `Vue 3` 项目
2. 安装 `vue-router`
3. 配置基础路由
4. 搭建移动端基础样式

### 第二阶段：Mock 数据接入

1. 建立景点与行程 Mock 数据文件
2. 封装简单请求函数，统一返回 Promise
3. 页面通过模拟接口读取数据

### 第三阶段：页面开发

1. 开发首页
2. 开发景点列表页
3. 开发景点详情页
4. 开发行程推荐页
5. 开发底部导航

### 第四阶段：联调与优化

1. 检查页面跳转逻辑
2. 检查移动端样式
3. 增加 loading 与空状态提示
4. 修复基础展示问题

## 13. 验收标准

满足以下条件即可视为首版完成：

1. 网站可在手机尺寸下正常浏览
2. 首页、景点列表、景点详情、推荐行程 4 个页面可正常访问
3. 页面数据来自本地 Mock
4. 至少包含 6 个北京景点
5. 点击景点可查看详情
6. 行程页至少展示 2 条推荐路线

## 14. 后续可扩展方向

如果后面需要继续迭代，可以追加以下功能：

1. 搜索景点
2. 收藏景点
3. 城市切换
4. 景点图文详情增强
5. 接入真实后端接口
6. 增加地图与路线导航能力
